# TechMarket Server Notes

## Назначение документа

Эта памятка объясняет этапы разработки серверной части TechMarket простым языком. Ее можно использовать как основу для диплома и как рабочую документацию для дальнейшей разработки.

Документ ведется накопительно: старые разделы не удаляются и не сокращаются, а новые итерации добавляются ниже. Так проще восстановить ход разработки и объяснить, почему архитектура стала именно такой.

## Что уже сделано

На первых этапах был подготовлен фундамент серверного приложения на NestJS:

- сервер запускается как REST API;
- все маршруты имеют общий префикс `/api`;
- подключена конфигурация через `.env`;
- подключен PostgreSQL через Prisma ORM;
- подключена автоматическая валидация входных данных;
- подключена Swagger-документация;
- добавлен health endpoint для проверки работоспособности сервера;
- добавлены категории каталога;
- добавлены шаблоны характеристик категорий;
- добавлены бренды и товары.

## Общая архитектура

Проект разделен на несколько частей:

| Часть | Путь | Назначение |
|---|---|---|
| Server | `apps/server` | Backend на NestJS |
| Prisma | `prisma` | Схема базы данных и миграции |
| Env | `.env` | Настройки окружения |
| Client | `apps/client` | Будущий frontend на React + Vite |

Главная идея архитектуры: frontend и backend развиваются отдельно. Клиент отправляет HTTP-запросы на сервер, сервер работает с базой данных и возвращает JSON.

Общая схема:

```text
Frontend React + Vite
        |
        | HTTP request
        v
NestJS Controller
        |
        v
NestJS Service
        |
        v
PrismaService
        |
        v
PostgreSQL
```

## Итерация 7: Избранное и сравнение товаров

На седьмом этапе добавлены два пользовательских модуля, которые нужны для современного интернет-магазина техники: избранное и сравнение товаров.

`WishlistModule` отвечает за избранные товары пользователя. Избранное хранится в базе данных, поэтому после входа в аккаунт пользователь видит один и тот же список на разных устройствах. Все endpoints избранного требуют авторизацию через JWT:

```text
Authorization: Bearer <accessToken>
```

Endpoints избранного:

```text
GET    /api/wishlist
POST   /api/wishlist
DELETE /api/wishlist/:productId
```

Пример добавления товара в избранное:

```json
{
  "productId": "product-id"
}
```

В базе для этого используется таблица `WishlistItem`. Она содержит `userId`, `productId` и даты создания/обновления. Для пары `userId + productId` добавлено уникальное ограничение, поэтому один и тот же пользователь не может добавить один товар в избранное несколько раз. Если повторно отправить запрос на добавление уже существующего товара, сервер вернет существующую запись, а дубль не создаст.

`ProductCompareModule` отвечает за сравнение товаров. В отличие от избранного, сравнение сделано stateless: сервер не хранит список сравнения в базе. Frontend просто отправляет массив id товаров, а backend строит таблицу сравнения по актуальным данным из базы.

Endpoint сравнения:

```text
POST /api/products/compare
```

Пример тела запроса:

```json
{
  "productIds": ["product-id-1", "product-id-2", "product-id-3"]
}
```

Правила сравнения:

- можно сравнивать от 2 до 3 товаров;
- все товары должны существовать;
- все товары должны быть активными;
- все товары должны относиться к одной категории;
- сравниваются только характеристики из `CategorySpecTemplate`, где `isComparable = true`.

Пример ответа:

```json
{
  "category": {
    "id": "category-id",
    "name": "Ноутбуки и компьютеры"
  },
  "products": [],
  "rows": [
    {
      "key": "ram",
      "label": "Оперативная память",
      "unit": "GB",
      "type": "NUMBER",
      "values": {
        "product-id-1": 16,
        "product-id-2": 32
      },
      "bestProductIds": ["product-id-2"]
    }
  ],
  "aiSummary": "Demo AI-анализ сформирован сервером по правилам, без внешнего платного API..."
}
```

Поле `rows` удобно напрямую использовать на frontend для таблицы сравнения. В каждой строке есть:

- `key` - технический ключ характеристики;
- `label` - понятное название характеристики для пользователя;
- `unit` - единица измерения, если она есть;
- `values` - значения по каждому товару;
- `bestProductIds` - товары, у которых значение характеристики считается лучшим.

Логика выбора лучшего значения:

- для `NUMBER` обычно больше значит лучше;
- для `weight`, `powerConsumption`, `noiseLevel` меньше считается лучше;
- для `BOOLEAN` значение `true` лучше `false`;
- для `STRING` и `SELECT` лучший вариант автоматически не выбирается;
- если значения равны, лучшими считаются все товары с этим значением.

`aiSummary` в этой итерации не использует внешний API и платные LLM-сервисы. Это демонстрационный AI-like анализ на основе правил: сервер смотрит цену, наличие, числовые характеристики и количество лучших параметров у каждого товара. Такой подход хорошо подходит для диплома, потому что показывает идею интеллектуальной рекомендации без зависимости от внешнего сервиса.

В дипломе можно написать:

> На этапе расширения пользовательских функций были реализованы избранное и сравнение товаров. Избранное хранится в базе данных и привязано к учетной записи пользователя. Сравнение товаров выполняется без сохранения состояния на сервере: клиент передает список товаров, а сервер строит таблицу по шаблонам характеристик выбранной категории. Дополнительно реализовано демонстрационное AI-like заключение, которое на основе правил анализирует цену, наличие и лучшие характеристики товаров. Это позволяет показать инновационный элемент системы без подключения платных внешних API.

## Итерация 6: Корзина и заказы

На шестом этапе добавляются корзина и оформление заказов. Это основной пользовательский сценарий интернет-магазина: выбрать товар, добавить его в корзину, оформить заказ и затем видеть историю заказов.

Корзина на сервере работает только для авторизованных пользователей. Гостевая корзина будет реализована позже на frontend через `localStorage`, а после входа пользователя ее можно будет синхронизировать с сервером.

Основные сущности:

| Сущность | Назначение |
|---|---|
| `CartItem` | Позиция товара в корзине пользователя |
| `Order` | Заказ пользователя с контактными данными и статусом |
| `OrderItem` | Товарная позиция внутри заказа |
| `OrderStatus` | Перечисление возможных статусов заказа |

Статусы заказа:

```text
NEW
PROCESSING
CONFIRMED
SHIPPED
COMPLETED
CANCELLED
```

Endpoints корзины:

```text
GET    /api/cart
POST   /api/cart
PATCH  /api/cart/:id
DELETE /api/cart/:id
DELETE /api/cart
```

Все endpoints корзины требуют авторизацию:

```text
Authorization: Bearer <accessToken>
```

Пример добавления товара в корзину:

```json
{
  "productId": "product-id",
  "quantity": 2
}
```

Логика корзины:

- нельзя добавить неактивный товар;
- нельзя добавить товар, которого нет в наличии;
- нельзя добавить количество больше остатка на складе;
- если товар уже есть в корзине, количество увеличивается;
- итоговая цена корзины считается по актуальной цене товара из базы.

Endpoints заказов пользователя:

```text
POST /api/orders
GET  /api/orders
GET  /api/orders/:id
```

Пример оформления заказа:

```json
{
  "customerName": "Иван Иванов",
  "customerPhone": "+375291112233",
  "customerEmail": "ivan@example.com",
  "city": "Минск",
  "deliveryAddress": "ул. Ленина, 10-15",
  "deliveryMethod": "courier",
  "paymentMethod": "cash_on_delivery",
  "comment": "Позвонить за час до доставки"
}
```

Административные endpoints заказов:

```text
GET   /api/admin/orders
GET   /api/admin/orders/:id
PATCH /api/admin/orders/:id/status
```

Пример изменения статуса:

```json
{
  "status": "PROCESSING"
}
```

Логика оформления заказа:

- заказ нельзя оформить с пустой корзиной;
- заблокированный пользователь не может оформить заказ;
- перед созданием заказа сервер повторно проверяет наличие товаров;
- заказ создается в транзакции Prisma;
- в этой же транзакции создаются позиции заказа;
- остатки товаров уменьшаются;
- корзина пользователя очищается;
- в `OrderItem.price` сохраняется цена товара на момент оформления заказа.

Почему используется транзакция:

```text
создать заказ
создать позиции заказа
уменьшить остатки
очистить корзину
```

Эти действия должны выполниться вместе. Если одно из них не удалось, остальные тоже не должны быть сохранены. Так база данных не окажется в противоречивом состоянии.

В дипломе можно написать:

> Для реализации процесса покупки были добавлены модули корзины и заказов. Корзина хранится на сервере для авторизованных пользователей и содержит позиции товаров с количеством. При оформлении заказа сервер проверяет наличие товаров, создает заказ и позиции заказа, уменьшает остатки на складе и очищает корзину. Все операции оформления выполняются в транзакции Prisma, что обеспечивает целостность данных. Пользователь может просматривать только собственные заказы, а администратор имеет доступ ко всем заказам и может изменять их статусы.

## Итерация 5: Пользователи, авторизация и OAuth

На пятом этапе добавляется модуль пользователей и авторизации. После этого сервер начинает различать гостя, обычного пользователя и администратора.

Главная идея:

- пользователь регистрируется по email и паролю;
- пароль не хранится в открытом виде, в базе лежит только `passwordHash`;
- после входа сервер выдает короткоживущий `accessToken`;
- `accessToken` отправляется клиентом в заголовке `Authorization`;
- `refreshToken` хранится в HTTP-only cookie и нужен для получения нового access token;
- администратор может управлять пользователями и назначать роль `ADMIN`;
- обычный пользователь не может сам стать администратором.

Схема токенов:

```text
POST /api/auth/login
        |
        v
accessToken -> JSON response
refreshToken -> HTTP-only cookie
```

Почему так удобно:

- access token легко тестировать через Postman и Swagger;
- refresh token не доступен JavaScript-коду в браузере;
- при истечении access token пользователь может получить новый без повторного ввода пароля;
- подход хорошо подходит для будущего frontend на React.

Endpoints авторизации:

```text
POST /api/auth/register
POST /api/auth/login
POST /api/auth/refresh
POST /api/auth/logout
GET  /api/auth/me
GET  /api/auth/google
GET  /api/auth/google/callback
GET  /api/auth/vk
GET  /api/auth/vk/callback
```

Пример регистрации:

```json
{
  "name": "Иван Иванов",
  "email": "ivan@example.com",
  "password": "Password123"
}
```

Пример входа:

```json
{
  "email": "admin@techmarket.local",
  "password": "Admin12345"
}
```

Пример успешного ответа:

```json
{
  "accessToken": "jwt...",
  "user": {
    "id": "user-id",
    "email": "admin@techmarket.local",
    "name": "TechMarket Admin",
    "role": "ADMIN"
  }
}
```

Защищенный запрос:

```text
Authorization: Bearer <accessToken>
```

Пользовательские endpoints:

```text
GET   /api/users/me
PATCH /api/users/me
```

Административные endpoints пользователей:

```text
GET   /api/admin/users
PATCH /api/admin/users/:id
PATCH /api/admin/users/:id/block
```

Как появляются администраторы:

- первый администратор создается через seed;
- новые администраторы назначаются только существующим администратором;
- регистрация обычного пользователя всегда создает роль `USER`;
- роль можно изменить через `PATCH /api/admin/users/:id`.

Тестовые пользователи:

```text
admin@techmarket.local / Admin12345
user@techmarket.local / User12345
```

OAuth через Google и VK:

- пользователь нажимает “Войти через Google” или “Войти через VK”;
- backend перенаправляет его на страницу провайдера;
- после подтверждения провайдер возвращает пользователя на callback endpoint;
- backend получает email и provider id;
- если пользователь уже есть по email, OAuth-аккаунт привязывается к нему;
- если пользователя нет, создается новый пользователь с ролью `USER`;
- если provider не вернул email, пользователь не создается.

Переменные окружения для OAuth:

```env
CLIENT_URL=http://localhost:5173

GOOGLE_CLIENT_ID=
GOOGLE_CLIENT_SECRET=
GOOGLE_CALLBACK_URL=http://localhost:5000/api/auth/google/callback

VK_CLIENT_ID=
VK_CLIENT_SECRET=
VK_CALLBACK_URL=http://localhost:5000/api/auth/vk/callback
```

Если ключи Google/VK не заполнены, OAuth endpoints возвращают понятную ошибку конфигурации. Email/password авторизация при этом работает.

После добавления авторизации административные действия каталога закрываются:

```text
POST/PATCH/DELETE categories -> только ADMIN
POST/PATCH/DELETE category specs -> только ADMIN
POST/PATCH/DELETE brands -> только ADMIN
POST/PATCH/DELETE products -> только ADMIN
```

Публичные `GET` endpoints каталога остаются доступными без входа.

В дипломе можно написать:

> В системе реализована ролевая модель доступа с ролями USER и ADMIN. Пароли пользователей хранятся только в виде bcrypt-хэша. Для авторизации используется JWT access token, а refresh token хранится в HTTP-only cookie, что повышает безопасность пользовательской сессии. Административные операции каталога защищены guard-ами NestJS и доступны только пользователям с ролью ADMIN. Дополнительно заложена архитектура OAuth-входа через Google и VK, позволяющая создавать или связывать внешние аккаунты с пользователем по email.

Controller принимает HTTP-запросы. Service содержит бизнес-логику. PrismaService выполняет запросы к базе данных PostgreSQL.

## Глобальный префикс /api

В коде используется:

```ts
app.setGlobalPrefix("api");
```

Это значит, что все endpoints начинаются с `/api`.

Пример:

```text
health -> /api/health
products -> /api/products
auth/login -> /api/auth/login
```

Так проще отделить backend API от frontend-страниц.

## CORS

CORS нужен, чтобы frontend мог обращаться к backend.

Разрешены адреса:

```text
http://localhost:5173
http://localhost:3000
```

`5173` - стандартный порт Vite. `3000` оставлен для совместимости с предыдущим прототипом.

Параметр `credentials: true` понадобится для авторизации через HTTP-only cookies.

## ValidationPipe

Глобальная валидация включена через `ValidationPipe`.

Настройки:

| Настройка | Что делает |
|---|---|
| `whitelist: true` | удаляет лишние поля из запроса |
| `forbidNonWhitelisted: true` | запрещает поля, которых нет в DTO |
| `transform: true` | преобразует значения к нужным типам |

Зачем это нужно:

- защита от лишних данных;
- меньше ошибок в сервисах;
- единая проверка форм регистрации, товаров, заказов;
- проще объяснить безопасность на защите.

## Prisma

Prisma используется как ORM.

ORM - это слой между кодом и базой данных. Вместо ручного SQL мы пишем TypeScript-код через Prisma Client.

Пример запроса:

```ts
this.prisma.product.findMany()
```

Это безопаснее и удобнее, чем вручную собирать SQL-строки.

## Prisma schema

Файл:

```text
prisma/schema.prisma
```

В нем описываются модели базы данных. Prisma на основе этого файла создает миграции и генерирует TypeScript-клиент.

Базовая настройка:

```prisma
generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
}
```

`generator client` говорит Prisma, что нужно сгенерировать TypeScript-клиент.

`datasource db` говорит, что используется PostgreSQL.

## PrismaService

Файлы:

```text
apps/server/src/prisma/prisma.module.ts
apps/server/src/prisma/prisma.service.ts
```

`PrismaService` делает Prisma доступной внутри NestJS.

В сервисах можно писать:

```ts
constructor(private prisma: PrismaService) {}
```

После этого service сможет работать с базой.

## Особенность Prisma 7

В проекте используется Prisma 7. Для нее нужен adapter PostgreSQL.

Поэтому установлен пакет:

```text
@prisma/adapter-pg
```

В `PrismaService` создается adapter:

```ts
const adapter = new PrismaPg({
  connectionString: configService.getOrThrow<string>("DATABASE_URL"),
});

super({ adapter });
```

Простыми словами: Prisma получает строку подключения из `.env` и через adapter подключается к PostgreSQL.

## Итерация 2: CategoriesModule

На втором этапе добавлен первый полноценный доменный модуль сервера - категории каталога.

Зачем нужны категории:

- они формируют структуру каталога магазина;
- к категории позже будут привязаны товары;
- для категорий позже будут создаваться шаблоны характеристик;
- дерево категорий понадобится для меню frontend.

Архитектура модуля:

```text
HTTP request
   |
   v
CategoriesController
   |
   v
CategoriesService
   |
   v
PrismaService
   |
   v
PostgreSQL
```

Файлы модуля:

```text
apps/server/src/categories/
├── categories.module.ts
├── categories.controller.ts
├── categories.service.ts
└── dto/
    ├── create-category.dto.ts
    ├── update-category.dto.ts
    └── find-categories.dto.ts
```

Главная идея:

- controller принимает HTTP-запрос;
- DTO проверяет входные данные;
- service содержит бизнес-логику;
- PrismaService выполняет запрос к PostgreSQL.

Endpoints:

```text
GET    /api/categories
GET    /api/categories/tree
GET    /api/categories/:id
POST   /api/categories
PATCH  /api/categories/:id
DELETE /api/categories/:id
```

Удаление категорий сделано мягким. Это значит, что строка не удаляется из базы физически. Вместо этого поле `isActive` становится `false`.

Почему это удобно:

- не ломаются будущие связи с товарами;
- категорию можно восстановить;
- история данных остается в базе.

В дипломе можно написать:

> Модуль категорий реализован с использованием стандартной архитектуры NestJS: контроллер отвечает за обработку HTTP-запросов, сервис содержит бизнес-логику, а доступ к данным выполняется через Prisma ORM. Категории поддерживают иерархическую структуру за счет связи parentId, что позволяет строить дерево каталога. Удаление реализовано как мягкое скрытие через поле isActive, чтобы не нарушать целостность связанных данных.

## Итерация 3: Шаблоны характеристик категорий

На третьем этапе добавлен модуль шаблонов характеристик категорий.

Идея модуля:

- администратор выбирает категорию;
- затем создает для нее список характеристик;
- при добавлении товара форма сможет использовать эти характеристики;
- сравнение товаров будет строиться по характеристикам с `isComparable = true`.

Пример для категории “Ноутбуки”:

| key | label | type | unit |
|---|---|---|---|
| `screenSize` | Диагональ экрана | `NUMBER` | дюйм |
| `processor` | Процессор | `STRING` |  |
| `ram` | Оперативная память | `NUMBER` | GB |
| `ssd` | Объем SSD | `NUMBER` | GB |

Поле `key` вводится на английском. Оно нужно для кода, API и хранения характеристик товара. Поле `label` отображается пользователю на русском языке.

Endpoints:

```text
GET    /api/categories/:categoryId/specs
POST   /api/categories/:categoryId/specs
PATCH  /api/categories/:categoryId/specs/:specId
DELETE /api/categories/:categoryId/specs/:specId
```

Пример создания характеристики:

```json
{
  "key": "ram",
  "label": "Оперативная память",
  "type": "NUMBER",
  "unit": "GB",
  "isRequired": true,
  "isComparable": true,
  "sortOrder": 10,
  "helpText": "Введите число без единицы измерения. Например: 16."
}
```

Типы характеристик:

| Тип | Когда использовать |
|---|---|
| `STRING` | Текст: процессор, цвет, операционная система |
| `NUMBER` | Число: объем памяти, диагональ, мощность |
| `BOOLEAN` | Да/нет: есть Wi-Fi, есть Bluetooth |
| `SELECT` | Выбор из списка, будет расширен позже |

В дипломе можно написать:

> Для повышения гибкости каталога реализован модуль шаблонов характеристик категорий. Он позволяет администратору самостоятельно задавать набор характеристик для каждой категории товаров. Такой подход упрощает добавление разных типов техники, так как ноутбуки, холодильники и телевизоры могут иметь разные наборы параметров. Отдельное поле `isComparable` позволяет заранее определить, какие характеристики будут участвовать в сравнении товаров.

## Итерация 4: Бренды и товары

На четвертом этапе добавлены бренды и товары. Это основа полноценного каталога интернет-магазина.

Зачем нужен отдельный `Brand`:

- бренд можно выбрать из справочника при создании товара;
- проще фильтровать товары по брендам;
- меньше риска опечаток, например `Samsung`, `Samsng`, `samsung`;
- в админ-панели можно будет управлять логотипами и описаниями брендов.

Зачем нужен `Product`:

- товар является основной сущностью интернет-магазина;
- карточка товара хранит цену, остаток, изображения и описание;
- товар связан с категорией и брендом;
- характеристики товара хранятся в гибком JSON-поле `specs`.

Связи в базе:

```text
Product  many -> 1 Category
```

```text
PATCH  /api/brands/:id
```

```json

  "items": [],

  "page": 1,
  "limit": 12,
  "pages": 0
```

```json
{
  "title": "Lenovo IdeaPad 5 16",
  "slug": "lenovo-ideapad-5-16",
  "sku": "NB-LEN-0001",

  "price": 2599.99,
  "oldPrice": 2899.99,
  "categoryId": "category-id",
  "brandId": "brand-id",
```

- `NUMBER` должен быть числом;
- `BOOLEAN` должен быть логическим значением;

В дипломе можно написать:

> На этапе реализации каталога были добавлены сущности бренда и товара. Товар связан с категорией и брендом, содержит цену, остаток, изображения и гибкий набор характеристик в формате JSON. Перед сохранением товара сервер проверяет характеристики по шаблону выбранной категории, что снижает вероятность ошибок при заполнении каталога и подготавливает систему к реализации сравнения товаров.

## Итерация 8: Загрузка изображений

На восьмом этапе добавлен базовый механизм загрузки изображений для административной части магазина. Он нужен, чтобы администратор мог загружать изображения товаров, логотипы брендов и картинки категорий, а затем использовать полученный URL в полях `images`, `logo` или `image`.

Загрузка файлов вынесена в отдельный `UploadsModule`. Это удобно архитектурно: работа с файлами не смешивается с логикой товаров, брендов и категорий.

Endpoint загрузки:

```text
POST /api/uploads/image
```

Доступ:

```text
Только ADMIN
Authorization: Bearer <accessToken>
Content-Type: multipart/form-data
```

Параметр query:

```text
type=products | categories | brands | general
```

Пример:

```text
POST /api/uploads/image?type=products
```

Тело запроса отправляется как `multipart/form-data` с полем:

```text
file
```

Пример ответа:

```json
{
  "url": "/uploads/products/1710000000000-file-id.webp",
  "filename": "1710000000000-file-id.webp",
  "originalName": "laptop.webp",
  "mimeType": "image/webp",
  "size": 125000
}
```

Ограничения:

- разрешены только `jpeg`, `png`, `webp`, `gif`;
- максимальный размер файла - 5 MB;
- файлы сохраняются локально в папку `/uploads`;
- публичный доступ к файлам идет по URL `/uploads/...`;
- сам endpoint загрузки закрыт для обычных пользователей.

Важная идея: база данных не хранит сам файл. В базе хранится только строка-URL, например `/uploads/products/file.webp`. Это нормальный подход для интернет-магазина: файл лежит в файловом хранилище, а сущность товара хранит ссылку на него.

В дипломе можно написать:

> Для работы с изображениями был реализован модуль загрузки файлов. Администратор может загрузить изображение через защищенный endpoint, после чего сервер сохраняет файл в локальное хранилище и возвращает URL. Этот URL используется в карточках товаров, категориях и брендах. Подход разделяет хранение файлов и бизнес-данных: база данных хранит только путь к изображению, а не бинарное содержимое файла.

## Итерация 9: Адреса пользователя

На девятом этапе добавлен `AddressModule`. Он нужен, чтобы пользователь мог заранее сохранить один или несколько адресов доставки и затем использовать их при оформлении заказа.

Основная идея: адрес принадлежит конкретному пользователю. `userId` не передается в body запроса, а берется из JWT-токена. Благодаря этому пользователь не может создать, изменить или удалить чужой адрес.

Модель базы данных:

```text
Address
- id
- userId
- label
- city
- street
- house
- apartment
- zipCode
- isDefault
- createdAt
- updatedAt
```

Endpoints:

```text
GET    /api/addresses
POST   /api/addresses
PATCH  /api/addresses/:id
PATCH  /api/addresses/:id/default
DELETE /api/addresses/:id
```

Все endpoints требуют авторизацию:

```text
Authorization: Bearer <accessToken>
```

Пример создания адреса:

```json
{
  "label": "Дом",
  "city": "Минск",
  "street": "ул. Ленина",
  "house": "10",
  "apartment": "15",
  "zipCode": "220030",
  "isDefault": true
}
```

Поведение:

- первый созданный адрес автоматически становится основным;
- если пользователь делает адрес основным, остальные его адреса перестают быть основными;
- пользователь может редактировать и удалять только свои адреса;
- если удалить основной адрес, сервер назначит основным самый новый оставшийся адрес;
- при оформлении заказа теперь можно передать `addressId`.

Checkout поддерживает два варианта:

```json
{
  "customerName": "Иван Иванов",
  "customerPhone": "+375291112233",
  "customerEmail": "ivan@example.com",
  "addressId": "address-id",
  "deliveryMethod": "courier",
  "paymentMethod": "cash_on_delivery"
}
```

или старый ручной формат:

```json
{
  "customerName": "Иван Иванов",
  "customerPhone": "+375291112233",
  "customerEmail": "ivan@example.com",
  "city": "Минск",
  "deliveryAddress": "ул. Ленина, 10-15",
  "deliveryMethod": "courier",
  "paymentMethod": "cash_on_delivery"
}
```

В дипломе можно написать:

> Для повышения удобства оформления заказов был реализован модуль адресов пользователя. Пользователь может хранить несколько адресов доставки, назначать основной адрес и использовать сохраненный адрес при создании заказа. Контроль доступа реализован на уровне сервиса: все операции выполняются только с адресами текущего авторизованного пользователя, что предотвращает доступ к чужим данным.

## Итерация 10: Отзывы и рейтинг товаров

На десятом этапе добавлен `ReviewsModule`. Он нужен, чтобы пользователи могли оставлять оценки и комментарии к товарам, а каталог и карточка товара могли показывать средний рейтинг.

Модель базы данных:

```text
Review
- id
- userId
- productId
- rating
- comment
- isActive
- createdAt
- updatedAt
```

Главное ограничение:

```text
@@unique([userId, productId])
```

Это означает, что один пользователь может оставить только один отзыв на один товар. Если мнение изменилось, пользователь должен обновить существующий отзыв, а не создавать новый.

Endpoints:

```text
GET    /api/products/:productId/reviews
POST   /api/products/:productId/reviews
PATCH  /api/reviews/:id
DELETE /api/reviews/:id
```

Публичный endpoint:

```text
GET /api/products/:productId/reviews?page=1&limit=10
```

Он возвращает список активных отзывов, пагинацию и сводку рейтинга:

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "limit": 10,
  "pages": 0,
  "rating": {
    "average": 4.5,
    "count": 2
  }
}
```

Создание отзыва требует авторизацию:

```text
Authorization: Bearer <accessToken>
```

Пример создания:

```json
{
  "rating": 5,
  "comment": "Хороший товар, быстрая доставка."
}
```

Правила:

- рейтинг должен быть от 1 до 5;
- заблокированный пользователь не может создавать отзывы;
- пользователь может обновлять только свой отзыв;
- пользователь может удалить только свой отзыв;
- администратор может удалить любой отзыв;
- удаление мягкое: `isActive = false`.

Список товаров и карточка товара теперь дополнительно возвращают:

```json
{
  "rating": {
    "average": 4.5,
    "count": 2
  }
}
```

В дипломе можно написать:

> Для повышения доверия к товарам был реализован модуль отзывов и рейтингов. Пользователь может оставить оценку от 1 до 5 и текстовый комментарий к товару. Сервер контролирует, чтобы один пользователь мог оставить только один отзыв на один товар, а при необходимости мог обновить его. Для карточки товара и каталога рассчитывается средний рейтинг и количество активных отзывов. Удаление отзывов выполнено мягко, что позволяет скрывать пользовательский контент без физического удаления записи из базы данных.

## Итерация 12: Промокоды и скидки

На двенадцатом этапе добавлен `PromoCodesModule`. Он нужен, чтобы администратор мог создавать скидочные промокоды, а пользователь мог проверить и применить промокод при оформлении заказа.

Модель базы данных:

```text
PromoCode
- id
- code
- description
- discountType
- value
- maxDiscount
- minOrderTotal
- usageLimit
- usedCount
- startsAt
- endsAt
- isActive
- createdAt
- updatedAt
```

В `Order` добавлены поля:

```text
promoCodeId
promoCodeCode
discountAmount
```

Это нужно, чтобы история заказа оставалась корректной даже после изменения или отключения промокода.

Endpoints администратора:

```text
GET    /api/admin/promo-codes
POST   /api/admin/promo-codes
PATCH  /api/admin/promo-codes/:id
DELETE /api/admin/promo-codes/:id
```

Все admin endpoints требуют:

```text
Authorization: Bearer <adminAccessToken>
```

Endpoint пользователя:

```text
POST /api/promo-codes/validate
```

Пример проверки промокода:

```json
{
  "code": "WELCOME10"
}
```

Пример ответа:

```json
{
  "code": "WELCOME10",
  "discountType": "PERCENT",
  "subtotal": 2599.99,
  "discountAmount": 260,
  "totalPrice": 2339.99
}
```

Checkout теперь дополнительно принимает:

```json
{
  "promoCode": "WELCOME10"
}
```

Правила промокодов:

- `PERCENT` считает скидку в процентах;
- `FIXED` считает фиксированную скидку;
- скидка не может быть больше суммы корзины;
- `maxDiscount` ограничивает максимальную скидку;
- `minOrderTotal` задает минимальную сумму заказа;
- `usageLimit` ограничивает количество использований;
- `usedCount` увеличивается только после успешного оформления заказа;
- удаление промокода мягкое: `isActive = false`.

В seed добавлены демонстрационные промокоды:

```text
WELCOME10
TECH50
```

В дипломе можно написать:

> Для реализации маркетинговых сценариев был добавлен модуль промокодов. Администратор может создавать процентные и фиксированные скидки, задавать период действия, минимальную сумму заказа и лимит использований. При оформлении заказа сервер проверяет промокод, рассчитывает скидку и сохраняет сумму скидки вместе с заказом. Это обеспечивает корректную историю заказов и позволяет использовать промокоды как часть бизнес-логики интернет-магазина.
## Итерация 13: Уведомления и mock email

На тринадцатом этапе добавлен `NotificationsModule`. Он нужен, чтобы сервер мог хранить уведомления пользователя и отдавать их будущему frontend-значку уведомлений на странице магазина.

Главная идея:

- реальные письма пока не отправляются;
- сервер создает запись уведомления в базе данных;
- рядом сохраняется `emailMockPayload`, то есть демонстрационный объект письма;
- `emailMockSent = true` показывает, что mock email был сформирован;
- mock email дополнительно логируется в консоль сервера.

В Prisma добавлены:

```text
NotificationType
Notification
User.notifications
```

Типы уведомлений:

```text
ORDER_CREATED
ORDER_STATUS_CHANGED
PROMO
SYSTEM
```

Модель `Notification` хранит:

```text
userId
type
title
message
isRead
emailMockSent
emailMockPayload
createdAt
readAt
```

Публичные endpoints пользователя:

```text
GET   /api/notifications
PATCH /api/notifications/:id/read
PATCH /api/notifications/read-all
```

Все endpoints требуют авторизацию:

```text
Authorization: Bearer <accessToken>
```

`GET /api/notifications` возвращает:

```json
{
  "items": [],
  "unreadCount": 0
}
```

`items` можно использовать для выпадающего списка уведомлений, а `unreadCount` - для числа на значке в шапке магазина.

`PATCH /api/notifications/:id/read` отмечает одно уведомление прочитанным. Пользователь может менять только свои уведомления. Если передать id чужого уведомления, сервер вернет `404`.

`PATCH /api/notifications/read-all` отмечает прочитанными все непрочитанные уведомления текущего пользователя и возвращает количество обновленных записей:

```json
{
  "updated": 3
}
```

В `NotificationsService` добавлен внутренний метод:

```ts
createUserNotification(...)
```

Этот метод используют другие модули. Сейчас он применяется в заказах:

- после успешного оформления заказа создается уведомление `ORDER_CREATED`;
- после изменения статуса заказа администратором создается уведомление `ORDER_STATUS_CHANGED`.

Важный момент архитектуры: уведомления создаются после успешной бизнес-операции. Если mock email или уведомление не создались, заказ уже не откатывается, потому что уведомление является вспомогательной функцией, а не основной частью покупки.

В seed добавлены два демонстрационных уведомления для:

```text
user@techmarket.local
```

В дипломе можно написать:

> Для повышения информативности личного кабинета был добавлен модуль уведомлений. Система хранит уведомления о создании заказа и изменении его статуса, а также формирует демонстрационную mock email-запись без подключения SMTP-сервера. Такой подход позволяет показать проектную готовность к email-уведомлениям и одновременно не усложнять дипломную реализацию внешними почтовыми сервисами. Frontend сможет использовать `unreadCount` для отображения индикатора новых уведомлений в шапке интернет-магазина.
## Итерация 14: E2E/API-тесты сервера

На четырнадцатом этапе добавлены e2e/API-тесты. Они проверяют сервер не по отдельным функциям, а как настоящий HTTP API: приложение NestJS поднимается в тестовом окружении, запросы отправляются через Supertest, а база данных работает через Prisma и PostgreSQL.

Для тестов используется отдельная база:

```text
techmarket_test
```

Это важно, потому что тесты создают пользователей, заказы, уведомления и меняют остатки товаров. Основная демонстрационная база `techmarket` при этом не затрагивается.

Добавлен пример тестовых переменных:

```text
.env.test.example
```

Главная переменная:

```env
DATABASE_URL_TEST="postgresql://postgres:postgres@localhost:5432/techmarket_test?schema=public"
```

Перед запуском тестов helper:

- создает `techmarket_test`, если базы еще нет;
- сбрасывает тестовую базу через Prisma migrations;
- запускает seed для заполнения тестовых данных;
- поднимает NestJS-приложение с теми же настройками, что и реальный сервер.

Чтобы настройки реального сервера и тестов не расходились, общая конфигурация вынесена в:

```text
src/bootstrap.ts
```

Там настраиваются:

- глобальный prefix `/api`;
- CORS;
- cookie parser;
- статическая раздача `/uploads`;
- глобальный `ValidationPipe`;
- Swagger.

Команда запуска:

```powershell
cd C:\TechMarket\apps\server
npm.cmd run test:e2e
```

Тесты проверяют:

- `GET /api/health`;
- login, register, refresh, logout и `GET /api/auth/me`;
- ошибку OAuth endpoints без Google/VK credentials;
- защиту admin endpoints: без token `401`, обычный user `403`, admin проходит;
- каталог товаров: пагинацию, поиск и получение товара по slug;
- корзину: добавление и обновление позиции;
- промокод `WELCOME10`;
- оформление заказа;
- уменьшение остатка товара после заказа;
- создание уведомления о заказе;
- изменение статуса заказа администратором;
- создание уведомления о смене статуса;
- запрет доступа к чужому уведомлению;
- сравнение товаров внутри одной категории;
- ошибку сравнения товаров из разных категорий.

В дипломе можно написать:

> Для проверки корректности серверной части были добавлены e2e/API-тесты. Тесты запускают NestJS-приложение в тестовом окружении, используют отдельную базу PostgreSQL `techmarket_test`, применяют миграции Prisma и выполняют HTTP-запросы к реальным endpoints. Такой подход позволяет проверить не только отдельные функции, но и полный пользовательский сценарий: авторизацию, работу каталога, корзину, оформление заказа, применение промокода, изменение статуса заказа и получение уведомлений.
## Итерация 15: Доставка, оплата, безопасность и CSV-обмен

На пятнадцатом этапе добавлены три доработки, которые приближают сервер к массовому использованию: справочники доставки и оплаты, базовое security hardening и экспорт/импорт данных для администратора.

### Справочники доставки и оплаты

Раньше заказ хранил `deliveryMethod` и `paymentMethod` как обычные строки. Теперь эти значения проверяются по справочникам:

```text
DeliveryMethod
PaymentMethod
```

Публичные endpoints для checkout:

```text
GET /api/delivery-methods
GET /api/payment-methods
```

Admin endpoints:

```text
GET    /api/admin/delivery-methods
POST   /api/admin/delivery-methods
PATCH  /api/admin/delivery-methods/:id
DELETE /api/admin/delivery-methods/:id

GET    /api/admin/payment-methods
POST   /api/admin/payment-methods
PATCH  /api/admin/payment-methods/:id
DELETE /api/admin/payment-methods/:id
```

Удаление справочников мягкое: запись не удаляется физически, а получает `isActive = false`.

В `Order` дополнительно сохраняются:

```text
deliveryMethodName
deliveryPrice
paymentMethodName
```

Это нужно, чтобы история заказа оставалась понятной даже после переименования способа доставки или оплаты в админке.

При checkout сервер:

- проверяет, что способ доставки активен;
- проверяет минимальную сумму заказа для доставки, если она задана;
- проверяет, что способ оплаты активен;
- считает промокод от суммы товаров;
- добавляет `deliveryPrice` к итоговой сумме заказа.

В seed добавлены:

```text
Delivery:
- courier
- pickup
- pickup_point

Payment:
- cash_on_delivery
- card_mock
- online_mock
```

### Security hardening

В сервер добавлены базовые защитные меры:

- отключен заголовок `X-Powered-By`;
- добавлены security headers;
- добавлен простой in-memory rate limit для `/api/auth/login`, `/api/auth/register`, `/api/auth/refresh`;
- добавлена runtime-проверка обязательных env-переменных.

Обязательные переменные:

```env
DATABASE_URL
JWT_ACCESS_SECRET
JWT_REFRESH_SECRET
```

JWT-секреты должны быть не короче 16 символов.

Rate limit сейчас демонстрационный и хранится в памяти процесса. Для production можно заменить его на Redis-based limiter, чтобы лимиты работали одинаково на нескольких инстансах сервера.

### CSV export/import

Добавлен `CatalogExchangeModule` для административной выгрузки и загрузки данных.

Admin endpoints:

```text
GET  /api/admin/export/products
GET  /api/admin/export/orders
POST /api/admin/import/products
```

Экспорт товаров возвращает CSV с колонками:


Экспорт заказов возвращает CSV с основными данными заказа, покупателя и списком позиций.

Импорт товаров работает по `sku`:

- если товар с таким `sku` уже есть, он обновляется;
- если товара нет, он создается;
- категория ищется по `categorySlug`;
- бренд ищется по `brandSlug`;

- характеристики проверяются по шаблону категории.

## Итерация 16: Подготовка Google OAuth к реальному подключению

На шестнадцатом этапе серверная часть Google OAuth подготовлена к реальному подключению через Google Cloud Console. Frontend в этой итерации не меняется.

Что уже было в сервере:

- `GET /api/auth/google`;
- `GET /api/auth/google/callback`;
- обмен authorization code на Google access token;
- получение профиля пользователя через Google userinfo;
- создание пользователя по email;
- привязка `OAuthAccount`;
- установка refresh cookie;
- редирект на `CLIENT_URL/auth/oauth-success`.

Что добавлено:

- OAuth `state` параметр;
- HTTP-only cookie `googleOAuthState`;
- проверка `state` в callback;
- очистка state cookie после callback;
- проверка `email_verified` от Google;
- `.env.example` с переменными для Google OAuth;
- e2e-проверка ошибки callback без state.

Переменные окружения:

```env
GOOGLE_CLIENT_ID=""
GOOGLE_CLIENT_SECRET=""
GOOGLE_CALLBACK_URL="http://localhost:5000/api/auth/google/callback"
CLIENT_URL="http://localhost:5173"
```

Как работает flow:

```text
1. Пользователь открывает /api/auth/google.
2. Сервер генерирует случайный state.
3. Сервер сохраняет state в HTTP-only cookie.
4. Сервер перенаправляет пользователя на Google OAuth consent screen.
5. Google возвращает пользователя на /api/auth/google/callback.
6. Сервер сравнивает state из URL и state из cookie.
7. Сервер меняет code на access token.
8. Сервер получает профиль Google.
9. Сервер проверяет, что email подтвержден.
10. Сервер создает или связывает пользователя.
11. Сервер ставит refresh cookie и отправляет пользователя на frontend success page.
```

Для Google Cloud Console нужно создать OAuth Client:

```text
Application type: Web application
Authorized redirect URI: http://localhost:5000/api/auth/google/callback
```

Для диплома можно написать:

> В системе реализована серверная интеграция с Google OAuth 2.0. Для защиты OAuth-flow используется параметр `state`, который сохраняется в HTTP-only cookie и проверяется при callback. После успешного входа сервер получает профиль Google, проверяет подтверждение email, создает или связывает пользователя с OAuth-аккаунтом и устанавливает refresh token в HTTP-only cookie. Такой подход позволяет подключить внешний сервис авторизации без хранения пароля пользователя в системе.
